# Maintenance Operations & Resource Management

This document outlines the core functional updates made to the Maintenance Module in PAMS. It highlights the architectural decisions, database optimizations, and system requirements (FR-4.1, FR-4.2, FR-4.3, FR-3.4) implemented to ensure efficient repair cycles and inventory oversight.

## 1. Shared Functionality & Inheritance (DRY Principle)

Following the project's requirement for code reusability (FR-4.2), we implemented a shared component architecture for ticket operations.

### Shared Dialog Components
Instead of duplicating the "Resolve" and "Assign" logic across different staff views, we created reusable dialog classes within `pages/maintenance_page.py`.
- **`ResolveTicketDialog`**: Used by both Maintenance Staff (to record work) and Administrators (for quality control/re-opening).
- **`AssignTicketDialog`**: A central component for staff allocation, ensuring that assignment logic (assignee validation, audit logging) is consistent across the entire application.

### Unified Ticket Query System
We deprecated legacy mock-data filters in favor of a dynamic, backend-driven query: `get_maintenance_tickets(status=None)`.
- **Justification**: This ensures that regardless of which dashboard is viewing the task (Maintenance, Admin, or Front-Desk), the source of truth is identical. It resolves apartment ties and reporter names seamlessly on the fly.

## 2. Maintenance-Exclusive Capabilities

The Maintenance dashboard manages the core repair lifecycle through specialized UI views and backend logic.

### Worker Load Balancing & Availability (FR-4.3)
To assist managers in fair task distribution, we implemented the **Worker Availability View**.
- **The "Left Join" Strategy**: The `get_worker_availability` function uses a `LEFT JOIN` between the `users` and `maintenance_tickets` tables.
- **Affordance**: This ensures that workers with **zero** active tasks still appear in the list with a count of `0`. A standard `JOIN` would hide available workers, creating a "Silent Failure" in the UI. 
- **Cognitive Load Reduction**: We implemented a visual threshold—workers with 3+ tasks automatically flip to a **"⏳ Busy"** (Orange) status, while others remain in a **"✅ Available"** (Green) state, allowing for instant decision-making.

### Real-Time Inventory Tracking (FR-3.4)
We implemented a dedicated **Equipment View** to manage the operational tools required for repairs.
- **Low-Stock Alerting**: The system monitors `quantity` levels. If stock falls below 5 units, the UI highlights the row in red.
- **Category Segmentation**: Inventory is programmatically split into "Tools," "Supplies," and "Parts" to allow for rapid navigation during emergency repair preparation.

## 3. Financial Analytics & Performance Metrics

To satisfy **FR-3.4** (Maintenance Financial Oversight) and **FR-4.1** (Task Lifecycle Tracking), we implemented live aggregation logic.

### Mathematical Accuracy (Julianday)
Instead of hardcoding "Average Resolution Time," we implemented a live SQL calculation:
```sql
SELECT AVG((julianday(resolved_at) - julianday(created_at)) * 24) 
FROM maintenance_tickets 
WHERE status IN ('resolved', 'closed')
```
- **Justification**: This provides a real-time performance indicator (KPI), measuring the efficiency of the repair team in hours.

### Financial Resilience (Coalesce)
When generating "Total Spend" or "Monthly Cost" metrics, we utilize `COALESCE(SUM(materials_cost), 0)`.
- **Justification**: This prevents the "NoneType" crash common in Python when an aggregation returns no rows. It ensures the dashboard reflects a clean `£0.00` instead of a system failure if no data exists for the current month.

## 4. Integration Guide (The Front-Desk Contract)

The Maintenance module acts as the "Consumer" of requests generated by the Front-Desk "Provider." To maintain system stability, the following integration contract must be followed.

### The Entry Point
New requests must be logged via the `log_maintenance_request` service:
```python
from database.db_service import log_maintenance_request

log_maintenance_request(
    apt_id="UUID", 
    description="Broken pipe", 
    priority="high" # 'high' triggers priority highlighting
)
```

### The "Vertical Slice" Simulation
To allow for parallel development while the Front Desk team completes their module, we implemented a **Debug Simulator** (`🛠️ Debug: Simulate Front Desk Request`).
- **Development Strategy**: This follows the "Vertical Slice" philosophy, allowing us to test the entire maintenance lifecycle (Report -> Assign -> Resolve -> Report) independently by programmatically injecting mock requests into the live database.

## 5. System Resilience & Verification

All core services are backed by automated verification to prevent regression.
- **Audit Logging**: Every assignment and resolution is natively logged to the `audit_log` table for accountability.
- **Unit Testing**: Run `pytest tests/test_maintenance.py` to verify the mathematical aggregations and worker availability logic.
